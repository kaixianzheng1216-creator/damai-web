# 大麦网 Web 端项目开发规范

本规范旨在规范前端开发流程、代码风格及技术选型，提升团队协作效率和代码质量。

## **1. 技术栈概览**

- **框架**: [Vue 3](https://vuejs.org/) (Composition API + `<script setup>`)
- **语言**: [TypeScript](https://www.typescriptlang.org/)
- **构建工具**: [Vite](https://vitejs.dev/)
- **UI 组件库**: [TDesign Vue Next](https://tdesign.tencent.com/vue-next/overview)
- **CSS 框架**: [Tailwind CSS v4](https://tailwindcss.com/)
- **状态管理**: [Pinia](https://pinia.vuejs.org/)
- **路由管理**: [Vue Router 4](https://router.vuejs.org/)
- **网络请求**: [Axios](https://axios-http.com/)
- **Mock 服务**: [MockJS](http://mockjs.com/) + [vite-plugin-mock](https://github.com/vbenjs/vite-plugin-mock)
- **代码校验/格式化**: [ESLint](https://eslint.org/), [Prettier](https://prettier.io/), [Oxlint](https://oxlint.dev/)

## **2. 命名规范**

### **2.1 目录与文件命名**

- **目录**: 采用 `kebab-case` (小写字母，中划线分隔)。例如：`src/components/user-profile/`
- **组件文件**: 采用 `PascalCase` (大驼峰)。例如：`UserProfile.vue`
- **普通 TS/JS 文件**: 采用 `kebab-case`。例如：`utils/date-helper.ts`
- **样式文件**: 采用 `kebab-case`。例如：`main.css`, `user-card.module.css`

### **2.2 组件命名**

- **组件名**: 至少包含两个单词（根组件 `App.vue` 除外），避免与 HTML 标签冲突。
- **引用组件**: 在模板中使用 `PascalCase` 或 `kebab-case`。推荐统一使用 `PascalCase`，如 `<UserProfile />`。

### **2.3 变量与函数命名**

- **变量/属性**: `camelCase` (小驼峰)。例如：`const userInfo = reactive({})`
- **函数/方法**: `camelCase`。例如：`const getUserData = () => {}`
- **常量**: `UPPER_SNAKE_CASE` (全大写，下划线分隔)。例如：`const MAX_RETRY_COUNT = 3`
- **类/接口/类型**: `PascalCase`。例如：`interface UserInfo {}`

## **3. 目录结构规范**

```text
src/
├── api/            # API 请求接口定义
├── assets/         # 静态资源（图片、字体、全局样式）
├── components/     # 全局公共组件
│   └── common/     # 基础/通用组件

├── utils/          # 工具函数
├── App.vue         # 根组件
└── main.ts         # 入口文件
```

## **4. Vue 开发规范**

### **4.1 脚本编写 (Script)**

- **必须使用 `<script setup lang="ts">`**。
- **Props 与 Emits**: 使用 `defineProps` 和 `defineEmits` 进行显式定义。
- **响应式**:
  - 优先使用 `ref` 定义基础类型和简单对象。
  - 只有在需要对一整组相关联的变量进行管理时才考虑使用 `reactive`。
- **解构**: 从 `store` 或 `props` 中解构响应式数据时，需使用 `storeToRefs` 或 `toRefs` 以保持响应性。

### **4.2 模板编写 (Template)**

- **简洁性**: 避免在模板中写复杂的逻辑表达式，应提取到 `computed` 或 `methods` 中。
- **指令**: 统一使用简写。例如：`:prop` 代替 `v-bind:prop`，`@click` 代替 `v-on:click`。
- **Key**: `v-for` 循环必须绑定唯一的 `key`，禁止使用 `index` 作为 key（除非数据是静态且无序变化的）。

### **4.3 样式编写 (Style)**

- **Tailwind CSS**: 优先使用 Tailwind CSS 进行样式开发，减少自定义 CSS。
- **Scoped**: 局部样式必须加上 `scoped` 属性。
- **TDesign 定制**: 如需覆盖 TDesign 样式，建议使用 CSS 变量或在 `ConfigProvider` 中配置，避免直接操作内部类名。

### **4.4 组件通信**

- **Props**: 仅用于父组件向子组件传递数据，子组件禁止直接修改 props。
- **Emits**: 子组件向父组件通信，必须使用 `defineEmits` 声明。
- **v-model**: 适用于自定义表单组件或需要双向绑定的简单状态，推荐使用 `v-model:xxx` 命名多个绑定。
- **Provide/Inject**: 仅在跨多层级组件通信时使用，需配合 TypeScript 的 `InjectionKey` 确保类型安全。

## **5. TDesign 与 Tailwind 使用建议**

- **布局**: 优先使用 TDesign 的 `Layout` 或 `Grid` 组件构建大框架。
- **细粒度调节**: 使用 Tailwind 的内联类处理间距 (`m-4`, `p-2`)、颜色 (`text-primary`) 和微调。
- **主题定制**:
  - TDesign 主题通过 `ConfigProvider` 进行全局控制。
  - Tailwind 主题在 `src/assets/main.css` 中的 `@theme` 块中扩展（Tailwind v4 规范）。

### **5.1 协同示例**

```vue
<template>
  <!-- 使用 TDesign 的卡片组件，配合 Tailwind 的内边距和圆角微调 -->
  <t-card class="m-4 shadow-lg rounded-xl overflow-hidden">
    <div class="flex items-center space-x-4">
      <!-- 使用 TDesign 的按钮，配合 Tailwind 的宽度控制 -->
      <t-button theme="primary" class="w-full h-12"> 确认预订 </t-button>
    </div>
  </t-card>
</template>
```

### **5.2 响应式设计**

- **大框架**: 使用 TDesign `t-col` 的 `span` 属性。
- **局部**: 使用 Tailwind 的响应式前缀（如 `hidden md:block`, `flex-col lg:flex-row`）。

## **6. API 调用与 Mock 规范**

- **统一封装**: 在 `src/api/` 下按模块创建接口文件。
- **类型定义**: 为每个接口的请求参数和返回数据定义 TypeScript 接口。
- **Mock 数据**:
  - 开发阶段在 `mock/` 目录下编写模拟接口。
  - 接口地址需保持与真实后端一致，通过 Vite 插件实现无感切换。

### **6.1 接口定义示例**

```typescript
// src/api/concert.ts
import axios from 'axios'
import type { ConcertInfo } from './types'

export const getConcertList = (params: { categoryId: string }) => {
  return axios.get<ConcertInfo[]>('/api/concert/list', { params })
}
```

### **6.2 错误处理与反馈**

- **全局拦截**: 在 Axios 拦截器中统一处理 401, 500 等通用错误。
- **业务反馈**:
  - 成功操作：使用 `Message.success()`。
  - 预期外错误：使用 `Message.error()` 或 `Notification`。
  - 加载状态：使用 TDesign 的 `t-loading` 组件或按钮的 `loading` 属性。

## **7. 性能优化与最佳实践**

- **计算属性**: 凡是依赖其他响应式数据计算得出的值，必须使用 `computed`。
- **监听器**: 避免滥用 `watch`，仅在数据变化需要触发副作用（如 API 请求）时使用。
- **条件渲染**:
  - 频繁切换显示隐藏使用 `v-show`。
  - 运行时不常改变的条件分支使用 `v-if`。
- **资源释放**: 在 `onUnmounted` 钩子中清理定时器、全局事件监听等。

## **8. 快速开始工作流 (小白必看)**

1. **新建页面**: 在 `src/views/` 下新建 `PascalCase.vue`。
2. **配置路由**: 在 `src/router/index.ts` 中注册新页面。
3. **定义接口**: 在 `src/api/` 中定义请求函数，并在 `mock/` 中编写模拟数据。
4. **编写 UI**: 使用 TDesign 组件库搭建骨架，用 Tailwind 润色样式。
5. **状态管理**: 如果数据需要跨页面共享，在 `src/stores/` 中定义 Pinia store。
6. **代码自检**: 运行 `npm run lint` 检查代码格式是否正确。

- [ ] 复杂的计算逻辑是否提取到了 `computed` 中？
- [ ] `v-for` 是否绑定了唯一的 `key`？
- [ ] 所有的 API 请求是否都定义了类型？
- [ ] 是否在需要的地方添加了加载中 (`loading`) 状态？
- [ ] 是否处理了可能的接口错误？
- [ ] 样式是否优先使用了 Tailwind CSS？
- [ ] 页面在不同屏幕尺寸下是否有基本的兼容性？

## **10. Git 提交规范**

提交信息格式：`<type>(<scope>): <subject>`

- **feat**: 新功能
- **fix**: 修补 bug
- **docs**: 文档变更
- **style**: 格式（不影响代码运行的变动）
- **refactor**: 重构（即不是新增功能，也不是修改 bug 的代码变动）
- **test**: 增加测试
- **chore**: 构建过程或辅助工具的变动

示例：`feat(user): 增加用户登录功能`
