# 前端代码规范

本项目基于 Vue 3 + TypeScript 架构，为了保证代码质量、提高团队协作效率并降低维护成本，制定了以下代码规范和开发指南。

## 🛠 核心工具链

- **ESLint**: 用于发现和修复 JavaScript/TypeScript/Vue 代码中的模式问题（Flat Config 模式）。
- **Oxlint**: 高性能的 Linter，主要用于快速捕获正确性相关的错误。
- **Prettier**: 代码格式化工具，保证代码风格一致。
- **TypeScript**: 强类型检查，编译前拦截大部分类型错误。
- **Husky & lint-staged**: Git Hooks，在提交代码前自动进行格式化和静态分析。

## 📏 提交流程与规范

### 1. 强制代码检查 (Core Rule)

> **⚠️ 每次修改代码后，必须运行 `npm run lint` 和 `npm run type-check` 以确保代码质量和类型安全。**

通过 Git Hooks (Husky) 已经配置了在提交（`git commit`）前拦截：

- 自动运行 `lint-staged` 对暂存区的文件执行 `oxlint --fix`, `eslint --fix` 和 `prettier --write`。
- 如果检查不通过，提交将会被终止。请根据终端报错信息修复问题后再提交。

### 2. 编辑器配置

推荐使用 **VS Code** 进行开发，并安装以下扩展：

- **Vue - Official (Volar)**: Vue 3 官方语言特性支持。
- **ESLint**: 代码质量检查。
- **Prettier - Code formatter**: 代码格式化。
- **EditorConfig for VS Code**: 自动应用 `.editorconfig` 规范。

项目根目录的 `.editorconfig` 定义了最基本的缩进规则：

- 缩进使用 **2 个空格** (`indent_size = 2`)
- 字符集 **UTF-8**
- 行尾使用 **LF**
- 最大行宽限制为 **100 字符** (配合 Prettier 的 `printWidth: 100`)
- 字符串推荐使用**单引号** (Prettier `singleQuote: true`)，并且**不使用分号结尾** (Prettier `semi: false`)。

## 💻 编码规范

### 1. 命名规范

- **组件名 (Components)**: 采用 **PascalCase** 大驼峰命名法。例如 `LoginForm.vue`, `PageContainer.vue`。
- **视图名 (Views)**: 采用 **PascalCase** 大驼峰命名法，且以 `View` 结尾。例如 `HomeView.vue`, `LoginView.vue`。
- **变量与函数**: 采用 **camelCase** 小驼峰命名法。例如 `userInfo`, `fetchData()`。
- **常量 (Constants)**: 采用 **UPPER_SNAKE_CASE** 全大写加下划线命名法。例如 `MAX_RETRY_COUNT`。
- **类名 (Classes) & 接口 (Interfaces)**: 采用 **PascalCase** 大驼峰命名法。
- **组合式函数 (Composables)**: 必须以 `use` 开头。例如 `useUserStore()`, `useAuth()`。

### 2. Vue 3 (Composition API) 规范

本项目**强制**使用 Vue 3 的 Composition API 和 `<script setup lang="ts">` 语法糖。**不推荐**使用 Options API。

#### 2.1 组件结构

在 SFC (Single-File Component) 中，保持以下代码块顺序：

```vue
<script setup lang="ts">
// 1. 引入依赖
// 2. 声明 props / emits
// 3. 响应式状态 (ref, reactive, computed)
// 4. 方法定义
// 5. 生命周期钩子 (onMounted 等)
// 6. 侦听器 (watch, watchEffect)
</script>

<template>
  <!-- 模板内容 -->
</template>

<style scoped>
/* 样式内容 (优先使用 TailwindCSS 实用类，尽量减少手写 CSS) */
</style>
```

#### 2.2 响应式状态

- 优先使用 `ref` 定义基本数据类型和对象，使用 `.value` 访问。
- 尽量保持状态的单一数据源，能用 `computed` 派生出来的数据，**绝对不要**用新的 `ref` 或 `reactive` 去存储。
- 避免在模板 (Template) 中进行复杂的逻辑计算，将其提取到 `computed` 中。

#### 2.3 组件拆分

保持组件的单一职责。如果一个组件满足以下任一条件，必须进行拆分：

- 既负责复杂的状态调度，又负责庞大的 UI 渲染。
- 包含 3 个以上的独立 UI 区块（例如表单、列表、侧边栏等）。
- 某段模板代码在其他地方可能会复用。
- 路由级别的视图组件（View）应该保持“薄”，主要用于组合业务组件和提供上下文，不要在 View 中编写大段的具体 UI 实现。

### 3. 自动导入 (Auto Import) 规范

本项目集成了 `unplugin-auto-import` 和 `unplugin-vue-components`。

- **无需手动 Import**: 常用的 Vue 核心 API (`ref`, `computed`, `watch`, `onMounted` 等)、Vue Router (`useRouter`, `useRoute`)、Pinia、VueUse 工具函数，以及 `useQuery`、`useMutation` 等，**均已配置自动导入**。
- **组件自动导入**: `src/components/` 目录下的组件和 TDesign Vue Next 的组件也会被自动解析和导入，直接在模板中使用即可（例如 `<t-button>` 或 `<page-container>`）。
- **类型支持**: 自动导入生成的类型声明在 `src/types/auto-imports.d.ts` 和 `src/types/components.d.ts` 中，已在 `tsconfig.app.json` 中配置，不会出现 TS 报错。

### 4. TypeScript 规范

- **禁止隐式 any**: 所有的变量、函数参数和返回值都应该有明确的类型定义。
- **类型复用**: 公共接口和类型统一放在 `src/types/` 或对应的 `src/api/types.ts` 中进行管理。
- **组件 Props**: 推荐使用基于类型的 Props 声明，并配合 `withDefaults` 设置默认值。
  ```ts
  interface Props {
    title: string
    isActive?: boolean
  }
  const props = withDefaults(defineProps<Props>(), {
    isActive: false,
  })
  ```

### 5. 样式与 CSS 规范

- **Tailwind CSS 优先**: 本项目使用 Tailwind CSS 框架，绝大部分样式应该通过原子类（Utility Classes）直接写在 `class` 中。
- **Scoped 样式**: 如果必须要写自定义 CSS，务必添加 `scoped` 属性，避免污染全局。
- **样式变量**: 主题色、字体等全局变量可以在 Tailwind 配置或 `src/styles/index.css` 中统一定义。
